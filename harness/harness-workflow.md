# Harness Workflow（ハーネスワークフロー）

> Claude Code に「複雑なタスクを安全・確実に進める手順」を与える定義ファイル。
> グローバル CLAUDE.md（`~/.claude/CLAUDE.md`）から参照されることで、複雑なタスクや「ハーネス開始」のキーワードで自動発動します。
>
> 原典: Anthropic "Harness Design for Long-Running Agentic Applications" / OpenAI "Harness Engineering"

---

## 適用条件

以下のいずれかに該当するとき、このワークフローで進める:

- 30分以上かかりそうな実装タスク
- 新機能開発・複雑なバグ修正・UI構築
- ユーザーから「**ハーネス開始**」と言われたとき

簡単な修正・質問・確認には不要（普通に対話で進めてよい）。

---

## 3層ワークフロー: Planner → Generator → Evaluator

「**生成と評価を分離する**」のが最大の原則。同じAIが書いて、同じAIが「OK」と言うのは検証になっていない。

### Layer 1: Planner（仕込み = 9割）

**まずユーザーに以下を確認する（必須3 + 任意2）:**

| # | 項目 | 質問 | 必須 |
|---|---|---|---|
| 1 | **ゴール** | 何を実現したいですか？ | 必須 |
| 2 | **背景・課題** | なぜ必要ですか？今どう困っていますか？ | 必須 |
| 3 | **完了イメージ** | どうなっていればOKですか？ | 必須 |
| 4 | **こだわり・制約** | デザイン、連携先、期限、避けたいことは？ | 任意 |
| 5 | **参考情報** | URL、スクショ、既存コード等は？ | 任意 |

**AI側で埋めること:**
- 技術スタック選定（環境・既存プロジェクトに最適なもの）
- 実装アプローチ・手順・優先順位
- スコープ（やること・やらないこと）
- 出力形式・検証方法

**未回答項目の扱い:**
「わからない」「おまかせ」と言われたら → AIがベストプラクティスで補完。最低限「ゴール」さえあれば仕様書を完成できる。

**Plannerの完了条件:**
仕様書をユーザーに提示し、**承認を得てから** Generator に進む。**Plan Mode の活用を推奨**。

---

### Layer 2: Generator（実装）

- 承認された仕様書のとおり一気に実装する
- 仕込みが十分なら追加質問は最小限
- 可能なら、テストを先に書いて失敗を確認してからコード実装

---

### Layer 3: Evaluator（検証）

**生成と評価は分離する。**

1. テスト実行で動作確認
2. UIがある場合はブラウザ等で実機確認
3. 完了イメージのチェックリストを1つずつ照合
4. 不合格 → 具体的フィードバックで Generator に修正指示

**失敗 → ルール変換:**

問題が見つかったら「気をつけて」で終わらせない。

1. 何が・なぜ起きたかを言語化
2. AIが再発防止ルールを具体化（禁止語・命名規則・制約等）
3. プロジェクトの `CLAUDE.md` または `.claude/rules/` に追記
4. 次回から自動適用

---

## 全体の流れ

```
ハーネス開始
  ↓
Planner: 5項目を確認 → AIが補完 → 仕様書をユーザーに提示
  ↓ (ユーザー承認)
Generator: 仕様書のとおり一気に実装
  ↓
Evaluator: テスト・実機確認・仕様照合
  ├─ 合格 → 完了
  └─ 不合格 → ルール化して Generator へ戻す
```

---

## 心構え（Anthropic公式より）

- **百科事典ではなく目次を渡す**: 指示ファイルは巨大マニュアルにせず、必要な情報だけ段階的に開示する
- **Human-in-the-loop**: Planner と Generator の間に必ず人間の承認を挟む
- **Independent verification**: Evaluator は Generator と同じ視点で検証しない
- **継続的改善**: 失敗を「気をつける」で終わらせず、必ず再発防止ルールに変換する

## 補足: Dynamic Workflows（最新・2026-06公式リリース）

> Anthropic公式記事: https://claude.com/blog/a-harness-for-every-task-dynamic-workflows-in-claude-code

このファイルが定義しているのは **静的ハーネス**（事前に決めたPlanner→Generator→Evaluatorの3層を1人のClaudeが順番に通す仕組み）です。
2026年6月、上位互換として **Dynamic Workflows** という新機能が公式リリースされました。

| | 「ハーネスやって」（このファイル） | 「ワークフローやって」（Dynamic Workflows） |
|---|---|---|
| 誰がやる | 1人の同じClaudeが頭の中で段階分け | Claude自身がJSを書いて別人格Claudeを複数organize |
| 並列性 | 直列のみ | 並列実行可（worktree隔離） |
| 検証の独立性 | 同じClaudeがEvaluatorなのでself-preferential biasの限界あり | 別contextのClaudeが検証 → 構造的に客観視 |
| トークン消費 | 1〜2倍 | 5〜10倍以上ありうる |
| いつ使う | **普段**（通常タスク） | **ここぞ**（長尺・徹底検証・並列で複数案・migration） |

**使い方**: 「ワークフローやって」「ultracode」と自然言語で依頼するだけ。JavaScriptを書く必要なし。Claude自身がその場でハーネスを書きます。

**使い分けの覚え方**:
- ハーネス = "失敗しない仕組み"（1人で丁寧に・普段使い）
- ワークフロー = "チームを組んで分業"（複数人で徹底的に・ここぞ）

**このファイルは温存**します。Dynamic Workflowsはこの静的ハーネスを置き換えるものではなく、**上位スイッチ**として並列に使い分けます。
